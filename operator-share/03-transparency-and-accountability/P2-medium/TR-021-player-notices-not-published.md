# TR-021: Player notice package ready for operator - not published or linked from product UI

| Field | Value |
|-------|-------|
| **Priority** | P2 - Medium |
| **Status** | Confirmed (disclosure package vs live product) |
| **Component** | Governance / publication |
| **CWE** | N/A |

## Summary

Plain-language **player notices and acknowledgement blocks** were prepared for operator use across briefs **01 - 04** and indexed in **05**. Live **ascendanteq.com** and **EQAscendant.exe** do **not** link or embed equivalent text at assessment time. Transparency work is **blocked on publication**, not only on drafting.

## Example

See [player-transparency-index-v1.md](../player-transparency-index-v1.md) for full list.

**Gap:** Materials exist in disclosure package - **not** confirmed live on operator site/patcher.

## Player scenario

**Context:** Player hears “transparency concerns” on forums.

**What happens:**

1. Checks official site - finds marketing, not notices.
2. Cannot validate or refute claims from **primary operator sources**.

## Implications

- Operator can adopt notices **as-is** or publish equivalents - reduces public advisory friction.
- Pairs with **TR-001**, **TR-010**.

## Remediation

1. Publish notices (or operator-edited versions) at stable URLs.
2. Link from patcher, Server Support, account pages.
3. Reply in [operator-response.md](../../operator-response.md) with live URLs + dates.

## Verification

- Each notice URL returns **200** and matches published date in changelog.
